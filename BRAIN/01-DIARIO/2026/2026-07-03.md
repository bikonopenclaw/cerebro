## Problemas

- `memory_search` permaneceu indisponível por quota insuficiente do provedor de embeddings; consolidação baseada em sessões visíveis, arquivos locais do Brain, diffs Git e inspeção direta.
- Execuções privadas de teste visual para relatórios operacionais falharam por limite de uso do modelo (`usage_limit_reached`), afetando resposta a pedido do Hebert sobre a falha naquele momento.
- Foram identificados PDFs já rastreados no Git do Brain, em desacordo com a regra atual de não versionar PDFs/artefatos binários; a correção foi remover esses arquivos do índice do Git e ampliar `.gitignore`, preservando cópias locais fora do versionamento.

## Decisões

- Relatórios de tickets NinjaOne não devem inferir KPIs quando a API não disponibilizar endpoint/permissão de listagem; registrar indisponibilidade é preferível a gerar métrica sem evidência.
- O Brain deve ignorar PDFs, imagens, ZIPs, sessões, caches e relatórios finais por padrão; exceções futuras exigem versão sanitizada textual e decisão explícita.

## Aprendizados

- Em WhatsApp atendimento, médias operacionais podem ser contaminadas por tickets antigos finalizados no dia; mediana, P90 e visão com/sem outliers são mais úteis para leitura executiva.
- Falhas por limite de uso do modelo podem impedir crons/testes privados sem indicar erro da integração final; devem ser registradas como pendência operacional de capacidade, não como falha funcional da API analisada.
