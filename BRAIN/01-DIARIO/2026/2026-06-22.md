## Problemas

- `memory_search` continuou indisponível por quota insuficiente do provedor de embeddings; consolidação baseada em memória local, arquivos e inspeção direta.
- Primeiro teste com dados reais da Alzira buscou e-mail financeiro do cadastro e também enviou para `adm@3oficiocariacica.com.br`; o script foi ajustado para, em modo de teste com destinatário explícito, não buscar cadastro automaticamente.
- SMTP DreamHost na porta 587 STARTTLS falhou com autenticação `535`; porta 465 SSL autenticou e foi adotada.

## Decisões

- A skill NFS-e da Bikon deve permanecer sem inscrição municipal configurada; ausência de IM não é pendência crítica enquanto a API aceitar.
- E-mail automático de NFS-e usa `fatura@bikontecnologia.com.br` como remetente e reply-to.
- Envio real para cliente externo continua exigindo autorização explícita do Hebert e trava no job, mesmo com SMTP funcionando.
- Quando houver duas ou mais NFS-e para o mesmo ID/documento de cliente, o padrão é um único e-mail por cliente, com todas as NFS-e, XMLs e boletos anexados.
- Grupo `relatórios operacionais` deve ser respondido visualmente como Kowalski, mas sem criar bot separado e sem permitir fala independente com Puppet Master ou outros agentes.
- Whisper local/faster-whisper é a rota padrão para áudios; Whisper API só deve ser usada se Hebert pedir explicitamente ou se a rota local falhar de forma não recuperável.

## Aprendizados

- Testes com dados reais de cliente exigem destinatário explícito e bloqueio de lookup automático de cadastro para evitar envio externo acidental.
- Em fluxo fiscal, e-mail ao cliente precisa carregar não só a nota, mas também o boleto relacionado quando a cobrança for por boleto.
- Um template HTML de e-mail deve ser multipart, com texto simples e HTML, para manter compatibilidade com clientes de e-mail.
- A memória operacional precisa registrar não só o que funciona, mas também quais portas/configurações falharam, para evitar regressão.
